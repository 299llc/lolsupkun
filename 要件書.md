# LoL Build Advisor — 要件定義書

**リアルタイムAIビルド提案 & マクロコーチングデスクトップアプリ**
合同会社299 | v0.4.0 — 2026年3月

> ★ v0.4.0 更新 — オブジェクト状況リアルタイム表示、観戦モード対応、サモナースペル/ルーン/スキルオーダー提案、Season2 patch25.09+タイマー修正、ブラウザテスト基盤追加、コアモジュール分離 (electron/core/)

---

## 1. プロダクト概要

### 1.1 目的

League of Legendsの試合中に、リアルタイムの戦況データ（KDA・CS・レベル・アイテム）を自動取得し、4種のClaude AI分析で最適なアイテムビルド・マクロ判断・対面対策・試合後評価を提供するデスクトップアプリ。

### 1.2 ターゲットユーザー

LoLプレイヤー（ランク問わず）。ビルド判断に時間をかけたくない、対面構成や戦況に応じたビルド判断を自動化したい、マクロ判断を改善したいユーザー。

### 1.3 プラットフォーム

| 項目 | 内容 |
|------|------|
| フレームワーク | Electron (Node.js + Chromium) |
| フロントエンド | React + Tailwind CSS v4 + Lucide Icons |
| データ取得 | Riot Live Client Data API (port 2999) |
| イベント検知 | LCU WebSocket (league-connect) |
| ビルドデータ | OP.GG (コアビルド・カウンターアイテム) |
| チャンプ・アイテムデータ | Data Dragon CDN (日本語対応) |
| AIエンジン | Claude API (Haiku / Sonnet) — 4種プロンプト |
| 対応 OS | Windows 10/11（macOSは将来対応） |

## 2. システム構成

### 2.1 アーキテクチャ（現行: BYOK型）

ユーザーが自身のAnthropic APIキーを設定するBYOK（Bring Your Own Key）型。バックエンドサーバー不要で運用コストゼロ。

```
┌─────────────────────────────┐
│  LoL Game Client (port 2999)│  ← Live Client Data API（REST・試合中のみ）
└──────────────┬──────────────┘
               │ HTTPS (localhost, 3秒ポーリング)
┌──────────────▼──────────────┐
│  Electron Main Process      │
│  - Game Poller (3s間隔)     │
│  - Context Builder          │──────┬──────────────────┐
│  - 差分検知・呼び出し制御    │      │ HTTPS             │ HTTPS
│  - Macro Orchestrator       │      │                   │
│  - IPC 通信                 │      │                   │
└──────────────┬──────────────┘      │                   │
               │ IPC          ┌──────▼──────────┐ ┌──────▼──────┐
┌──────────────▼──────────────┐│ Claude API      │ │  OP.GG      │
│  Renderer (React UI)        ││ - Haiku (3種)   │ │  - コアビルド│
│  - チャンプセレクト画面      ││ - Sonnet(コーチ)│ │  - カウンター│
│  - チーム一覧 + KDA         │└─────────────────┘ └─────────────┘
│  - コアビルド表示            │
│  - AIアイテム提案            │      ┌─────────────────────┐
│  - マクロアドバイス          │      │  Data Dragon CDN    │
│  - マッチアップTip           │      │  - チャンプ情報      │
│  - 試合後コーチング          │      │  - アイテム情報      │
│  - デバッグパネル            │      │  - スキル説明(日本語)│
└─────────────────────────────┘      └─────────────────────┘

┌─────────────────────────────┐
│  LCU WebSocket              │  ← 試合開始/終了/チャンプセレクト検知
│  (lockfileからポート取得)    │     league-connect パッケージ
└─────────────────────────────┘
```

| レイヤー | 技術 | 役割 |
|----------|------|------|
| Data Source | Riot Live Client Data API | 試合中のリアルタイムデータ提供 (localhost:2999) |
| Event Bus | LCU WebSocket | 試合開始/終了/チャンプセレクトの検知 |
| Build Data | OP.GG | チャンピオン別コアビルド・カウンターアイテム取得 |
| Champion Data | Data Dragon CDN | チャンプ情報・アイテム情報・スキル説明（日本語） |
| Main Process | Electron (Node.js) | ポーリング・Context Builder・AI呼び出し制御・IPC通信 |
| AI Engine | Claude API (Haiku + Sonnet) | 4種の分析（アイテム提案/マッチアップ/マクロ/コーチング） |
| Renderer | React + Tailwind CSS v4 | 全画面UI |

### 2.3 データフロー

#### チャンプセレクト時
1. LCU WebSocketがチャンプセレクト開始を検知
2. LCU APIからピック情報を取得（味方・敵チャンプ）
3. OP.GGからコアビルド・スキルオーダーを取得
4. Data Dragonからチャンプ情報・スキル説明を取得
5. チーム構成分析（AD/AP比率、CC有無、回復有無等）をUIに表示
6. コアビルドをRendererに送信・表示

#### 試合中
1. Main ProcessがLive Client Data APIへのポーリング開始（3秒間隔）
2. `/liveclientdata/allgamedata` から全プレイヤーデータ取得
3. 差分検知: 前回データと比較し、AI呼び出しトリガーを判定
4. Context Builder: 静的コンテキスト（構成・特性）＋動的コンテキスト（Lv・アイテム・キル差）を構築
5. 4種のAIを状況に応じて呼び出し:
   - **アイテム提案** (Haiku): 差分検知トリガー時
   - **マッチアップTip** (Haiku): 試合開始時1回
   - **マクロアドバイス** (Haiku): 2分間隔 + オブジェクト取得時
   - **コーチング** (Sonnet): 試合終了時1回
6. Rendererにデータ送信・表示

### 2.4 AI判断アーキテクチャ

#### 4種のAIプロンプト

| AI機能 | モデル | タイミング | 目的 |
|--------|--------|-----------|------|
| アイテム提案 (SYSTEM_PROMPT) | Haiku | 差分検知トリガー | 戦況に応じた次のアイテム5つを提案 |
| マッチアップTip (MATCHUP_PROMPT) | Haiku | 試合開始時1回 | 対面チャンプとの戦い方を3行で解説 |
| マクロアドバイス (MACRO_PROMPT) | Haiku | 2分間隔+オブジェクト時 | 今すべき行動を指示（オブジェクト優先） |
| コーチング (COACHING_PROMPT) | Sonnet | 試合終了時1回 | KDA・試合展開の総合評価 |

#### Context Builder（コンテキスト構築）

静的情報（試合中不変）と動的情報（毎回変化）を分離して構築。

**静的コンテキスト:**

| 要素 | データソース | 判断への影響 |
|------|-------------|-------------|
| 自チャンプ・ポジション | activePlayer | ビルドパスの方向性 |
| 敵チーム構成 | allPlayers + Data Dragon | AP/AD比、カウンタービルド |
| 敵チャンプ特性 | スキル説明テキスト解析 | 回復/CC/シールドの有無 |
| 味方チーム構成 | allPlayers | シナジー・バランス判断 |

**動的コンテキスト:**

| 要素 | データソース | 判断への影響 |
|------|-------------|-------------|
| Lv・アイテム進捗 | activePlayer | 購入可能アイテムの絞り込み |
| 敵AD/AP比率 | チャンプ情報のattack/magic値 | 防御アイテム選択 (MR vs Armor) |
| KDA・キル差 | allPlayers.scores | 攻撃的 vs 防御的ビルド |
| 試合時間・フェーズ | gameData.gameTime | 序盤/中盤/終盤の判断 |

#### チャンプ特性自動検出

Data Dragonのスキル説明テキスト（日本語）から正規表現で特性を自動抽出:

| 特性 | 検出キーワード | 用途 |
|------|--------------|------|
| 回復 (Heal) | 回復、ヒール、ライフスティール | 回復阻害アイテム提案 |
| CC | スタン、スネア、ノックアップ等17種 | QSS・テナシティ提案 |
| シールド | シールド | シールド貫通提案 |
| Tank | Data Dragon tags | 貫通アイテム提案 |
| Fed | キル数5以上 | 脅威レベル表示 |

#### アイテム提案の3段階判断

AIが試合展開を見て攻守バランスを自動調整:

| 状態 | 条件 | 方針 |
|------|------|------|
| 優勢 | キル差+3以上 or レベル先行 | 火力アイテムでスノーボール加速 |
| 拮抗 | キル差±3以内 | コアビルド完成優先＋カウンター |
| 劣勢 | キル差-3以下 | 防御アイテムで生存力確保 |

#### マクロアドバイスのオブジェクトタイマー

ゲーム内イベントからオブジェクトの状態を秒単位で追跡:

| オブジェクト | 初回スポーン | リスポーン | 特記 |
|-------------|------------|-----------|------|
| ドラゴン | 5:00 (300s) | 5分 (300s) | — |
| バロン | 20:00 (1200s) | 6分 (360s) | — |
| ヴォイドグラブ | 8:00 (480s) | — | 14:45で消滅。リスポーンなし (Season2 patch25.09+) |
| ヘラルド | 15:00 (900s) | — | 20:00でバロン出現により消滅。リスポーンなし (Season2 patch25.09+) |

マクロAIには「スポーン済み・取得可能」「リスポーンまでX:XX」の状態を提供。スポーン済みオブジェクトは最優先で言及するルール。

#### Claude API呼び出しトリガー（差分検知）

| トリガー | 検知方法 | 対象AI | 想定頻度 |
|----------|---------|--------|---------|
| アイテム購入 | items配列の変化 | アイテム提案 | 1試合10-15回 |
| キル/デス発生 | scores変化 | アイテム提案 | 1試合10-20回 |
| 定期更新 | 2分経過 | アイテム提案+マクロ | 1試合約15回 |
| 試合開始直後 | 初回データ取得 | アイテム提案+マッチアップTip | 1回 |
| オブジェクト取得 | DragonKill/BaronKill等イベント | マクロ | 1試合5-10回 |
| 試合終了 | GameEnd or ポーリング停止 | コーチング | 1回 |

**デバウンス:**
- アイテム提案: 10秒以内の重複トリガーは1回に集約
- マクロ（通常）: 2分間隔
- マクロ（オブジェクト）: 30秒デバウンス

#### コスト見積もり

| 項目 | 値 |
|------|-----|
| アイテム提案 (Haiku) | 約15回/試合 × $0.002 = $0.03 |
| マッチアップTip (Haiku) | 1回/試合 × $0.002 = $0.002 |
| マクロアドバイス (Haiku) | 約20回/試合 × $0.002 = $0.04 |
| コーチング (Sonnet) | 1回/試合 × $0.01 = $0.01 |
| **合計コスト/試合** | **約$0.08** |
| **コスト/月 (1日5試合)** | **約$12** |

> ※ Prompt Caching活用時は入力コスト90%削減可能。

## 4. 利用API仕様

### 4.1 Live Client Data API（ポート2999）

| エンドポイント | 取得データ | 備考 |
|--------------|-----------|------|
| `/allgamedata` | 全データ（統合エンドポイント） | メインで使用 |
| `/activeplayer` | 自プレイヤーのステータス・ルーン・ゴールド | championStatsが豊富 |
| `/playerlist` | 全10人のチャンピオン・KDA・アイテム・レベル | scoresオブジェクトにKDA/CS |
| `/eventdata` | キル・ドラゴン・タワー等のイベント | オブジェクトタイマーに使用 |
| `/gamestats` | 試合時間・マップ・ゲームモード | gameTimeでタイマー表示 |

**制約事項:**

- 試合中のみアクセス可能（ゲーム外はConnectionError）
- APIキー不要・localhost限定 (127.0.0.1)
- 自己署名SSL証明書のため `rejectUnauthorized: false` で接続
- WebSocket/Streaming非対応（RESTのみ → ポーリング必須）

### 4.2 LCU WebSocket

| 項目 | 内容 |
|------|------|
| ポート | ランダム（lockfileから取得） |
| 認証 | Basic認証（lockfileのパスワード） |
| プロトコル | WAMP 1.0 over WSS |
| 主な用途 | 試合開始/終了検知、チャンピオン選択監視 |
| npmパッケージ | league-connect |

### 4.3 Claude API（4種のAI分析）

| 項目 | アイテム提案 | マッチアップTip | マクロアドバイス | コーチング |
|------|------------|---------------|----------------|-----------|
| モデル | Haiku (`claude-haiku-4-5-20251001`) | Haiku | Haiku | Sonnet (`claude-sonnet-4-6-20260217`) |
| max_tokens | 600 | 700 | 300 | 1500 |
| temperature | 0 | 0 | 0 | 0.3 |
| タイムアウト | 30秒 | 20秒 | 20秒 | 60秒 |
| レスポンス | JSON (items, reasoning) | JSON (summary, details) | JSON (advice, actions) | JSON (scores, sections) |

### 4.4 Data Dragon CDN

チャンピオン・アイテム・スキル情報の取得先。起動時にパッチバージョンを確認し、ローカルキャッシュと照合。

- バージョン取得: `https://ddragon.leagueoflegends.com/api/versions.json`
- チャンプ一覧: `https://ddragon.leagueoflegends.com/cdn/{ver}/data/ja_JP/champion.json`
- チャンプ詳細: `https://ddragon.leagueoflegends.com/cdn/{ver}/data/ja_JP/champion/{name}.json`
- アイテム一覧: `https://ddragon.leagueoflegends.com/cdn/{ver}/data/ja_JP/item.json`
- 画像: `https://ddragon.leagueoflegends.com/cdn/{ver}/img/champion/{name}.png`

キャッシュ先: `userData/patch-cache/` (champions.json, items.json, spells.json, patch-meta.json)

### 4.5 OP.GG

| 項目 | 内容 |
|------|------|
| 取得データ | チャンピオン別コアビルド（ポジション別）、カウンターアイテム |
| タイミング | チャンプセレクト時 or 試合開始時にチャンプ確定後 |
| キャッシュ | 同一チャンプ・ポジションは試合中再取得しない |

## 5. 非機能要件

| カテゴリ | 要件 |
|----------|------|
| パフォーマンス | CPU使用率 <3%、メモリ 100MB以下、ポーリング間隔 3秒、Claude API応答 30秒以内 |
| セキュリティ | ゲームデータ取得はlocalhost限定。APIキーはファイルベース保存 (`userData/.api-key`)。Main Process経由のみアクセス（Rendererから直接不可）。`contextIsolation: true`、`nodeIntegration: false` |
| UX | オーバーレイ表示 (alwaysOnTop切替可)。コンパクトサイズ (420x700px)。ダークテーマ。フレームレス・ドラッグ可能。ウィンドウ位置記憶 |
| Riot規約遵守 | 公式APIのみ使用。ゲーム内操作の自動化は行わない。配布時はRiot Developer Portalにアプリ登録必須 |
| Vanguard互換性 | Riot Vanguard（カーネルレベルアンチチート）との互換性を実機テストで検証済み |

## 10. リスクと対策

| リスク | 影響度 | 対策 |
|--------|--------|------|
| RiotがLocal APIを廃止/変更 | 高 | Riot Developerコミュニティを監視。APIレイヤーで切り替え容易に |
| LoLパッチでデータ構造変更 | 中 | Data Dragonバージョン自動検出・キャッシュ更新機能で対応 |
| Electronオーバーレイがフルスクリーンで表示されない | 高 | ボーダーレスウィンドウモード推奨。セカンドモニター対応 |
| VanguardがElectronをブロック | 高 | 実機テストで検証済み（現時点で問題なし）。ブロック時はセカンドモニター専用モード |
| Claude API応答遅延・障害 | 中 | タイムアウト設定。前回提案を維持。コアビルド（OP.GG）はAI不要 |
| Claude APIコスト超過 | 中 | Haiku主体でコスト抑制。差分検知+デバウンスで呼び出し制御 |
| OP.GGスクレイピング変更 | 中 | OP.GGのHTML構造変更時はパーサー更新が必要 |
| ToS違反のリスク | 高 | 公式APIのみ使用。ゲーム内操作の自動化は行わない |
