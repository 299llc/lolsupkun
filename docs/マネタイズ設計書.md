# マネタイズ設計書

## 1. 概要

ローカルLLM（無料）とクラウドAI（有料）の2層構造で、Stripe決済 + ライセンスキー方式でマネタイズする。ログイン不要。インフラはAWS CDKで一括管理。

### ライセンスキー方式の特徴

- ログイン・アカウント登録不要（キーを貼るだけ）
- キー形式: `RSPO-XXXX-XXXX-XXXX`
- 同時1デバイス制限（起動時にハートビートでデバイスID記録）
- 月次でStripe Subscriptionと同期（解約→キー無効化）

## 5. AWSインフラ構成

### アーキテクチャ

```
API Gateway (REST API)
  ├─ POST /webhook        → stripe-webhook Lambda
  ├─ POST /verify         → license-verify Lambda
  └─ GET  /portal/{key}   → stripe-portal Lambda

DynamoDB (licenses テーブル)
  PK: licenseKey
  GSI: stripeCustomerId-index
  属性:
    - licenseKey: string        (RSPO-XXXX-XXXX-XXXX)
    - email: string
    - stripeCustomerId: string
    - stripeSubscriptionId: string
    - plan: string              (pro | pro_plus)
    - status: string            (active | cancelled | expired)
    - deviceId: string | null
    - lastHeartbeat: number     (Unix timestamp)
    - gamesUsed: number         (月間使用試合数)
    - gamesLimit: number        (50 or -1)
    - createdAt: string
    - expiresAt: string | null

Secrets Manager
  - stripe-secret-key
  - stripe-webhook-secret

SES (メール送信)
  - ライセンスキー送付
  - 更新通知
```

### Lambda関数

#### stripe-webhook
- トリガー: Stripe Webhook
- イベント:
  - `checkout.session.completed` → キー生成、DynamoDB保存、メール送信
  - `customer.subscription.deleted` → キーのstatusを`cancelled`に更新
  - `invoice.payment_failed` → 猶予期間設定

#### license-verify
- トリガー: アプリからPOST
- リクエスト: `{ key, deviceId }`
- レスポンス: `{ valid, plan, gamesRemaining }` or `{ valid: false, reason }`
- ロジック:
  1. DynamoDBからキー検索
  2. status確認（active以外→無効）
  3. デバイスID確認（別デバイス→無効 or 上書き）
  4. 試合数確認（上限超過→無効）
  5. ハートビート更新

#### stripe-portal
- トリガー: ブラウザからGET
- ライセンスキーからStripe Customer IDを引き、Customer Portal URLを生成してリダイレクト
- ユーザーは自分で解約・カード変更が可能

## 7. アプリ側の変更点

### preload.js（追加IPC）

- `license:verify` → Lambdaにキー検証リクエスト
- `license:activate` → キー保存 + Bedrock有効化
- `license:deactivate` → キー削除 + ローカルに戻す

### SettingsDialog.jsx

- 既存のプロバイダー選択UIを拡張
- 「クラウド(Pro)」選択時にキー入力欄を表示
- 未認証時は「Proにアップグレード」ボタン → openExternal()でLP

### main.js

- 起動時にキー検証（キャッシュあり、24時間有効）
- 試合開始時にgamesUsedをインクリメント
- オフライン時は前回の検証結果をキャッシュから読む

## 9. 不正利用対策

| リスク | 対策 |
|--------|------|
| キー共有 | 同時1デバイス制限（deviceId記録） |
| デバイス偽装 | ハートビート間隔チェック（5分以内に別deviceId→ブロック） |
| 試合数水増し | サーバー側でカウント、アプリ側は参照のみ |
| キー総当たり | API Gatewayのスロットリング + キー形式の十分なエントロピー |

初期段階では過剰対策不要。ユーザー規模に応じて段階的に強化する。
