# ろるさぽくん — 基本設計書

## 前提条件

- Windows デスクトップアプリ（Electron + React + shadcn/ui + Tailwind CSS）
- LoLの試合中にリアルタイムでAIコーチングを提供する

---

## 1. 解くべき問題と5つの機能

| # | 機能 | 解決する問題 | いつ動く |
|---|------|-------------|---------|
| 1 | アイテム提案 | 敵構成に応じたアイテム選択がわからない | 試合中（状況変化時） |
| 2 | 対面マッチアップTip | 知らない対面で何に気をつけるべきかわからない | 試合開始時（ポジション確定後） |
| 3 | マクロアドバイス | ゲーム中盤以降、どこにいて何をすべきかわからない | 試合中（定期的） |
| 4 | ルールベースアラート | ファーム効率低下やオブジェクトタイミングを見逃す | 試合中（常時） |
| 5 | 試合後コーチング | 何が良くて何が悪かったかわからない | 試合終了後 |

  │ LoLクライアント │  │ AIサービス  │ │ ビルドデータ │

## 3. Main Process の3つの役割

### 3.1 データ収集

LoLクライアントが試合中に公開するローカルAPI（Live Client Data API）から、数秒間隔でゲームデータを取得する。

取得できる情報：
- 全プレイヤーのチャンピオン・レベル・アイテム・KDA・CS
- 自プレイヤーの現在ゴールド
- ゲーム内時間
- イベントログ（キル、オブジェクト獲得等）

加えて、LCU（League Client Update）APIからチャンプセレクト中のピック情報・ルーン・サモナースペルも取得する。

#### 対面チャンピオンの判定

チャンプセレクト時点では敵のロール割り当てが確定しない（フレックスピック・ロールスワップがあり得る）。Live Client Data APIは試合開始後に各プレイヤーのポジション情報を返すため、**対面の特定は試合開始時に行う**。マッチアップTipの生成もこのタイミングで実行する。

#### Live Client Data API で取得できないもの

- **他プレイヤーのゴールド**: 自分以外のゴールドは取得不可。アイテム価格の合算で推定する
- **レーン状態（laneState）**: APIにはウェーブ位置の情報がない。直近のキル/デス/タワー破壊イベントの発生レーンから簡易推定する。精度は低いため、マクロアドバイスでは補助情報として扱う
- **ドラゴン取得チーム**: イベントログのDragonKillイベントにKillerNameが含まれるため、プレイヤーのチームと照合して判定する

### 3.2 判断生成

集めたデータを元に、2つの方法でアドバイスを生成する。

**AI分析（機能1, 2, 3, 5）**
- ゲーム状況から事実を抽出・構造化し（前処理層）、LLMには構造化JSONで候補の優先順位付けと説明文生成だけを任せる（推論層）。詳細はセクション9参照
- 毎秒呼ぶわけにはいかないので、「状況が変わったとき」だけ呼ぶ仕組みが必要
- 外部のビルド統計データ（OP.GG等）をベースラインとして渡すことで、AIには「この試合固有の判断」だけに集中させる

**ルールベース判定（機能4）**
- AIを使わない即時判定。閾値ベースの単純なif文
- CS/分が一定以下なら警告、オブジェクトスポーンN秒前に予告、等
- AIの応答を待たずに瞬時に出せるのが強み

### 3.3 アプリ管理

Electron固有の仕事。ウィンドウ管理（フレームレス・常時最前面）、設定の永続化、APIキーの管理など。

## 4. Renderer（React UI）の画面設計

### 4.1 フェーズに応じた画面切替

LoLの試合フローに合わせて画面を切り替える。

```
待機中 ──► チャンプセレクト ──► 試合中 ──► 試合終了
                                          │
                                          └──► 待機中（次の試合へ）
```

| フェーズ | 表示内容 |
|---------|---------|
| 待機中 | 接続状態、設定画面 |
| チャンプセレクト | チーム構成、ルーン・スペル情報（LCU APIから取得） |
| 試合中 | **5機能のうち4つ**（アイテム提案、マッチアップTip、マクロアドバイス、アラート） |
| 試合終了 | 試合後コーチング |

### 4.2 試合中画面のレイアウト方針

「チラ見でわかる」が最重要。ゲーム中に読み込む余裕はない。

- **1つのアドバイスは2-3行以内**に収める
- 優先度の高い情報を上に、補足は折りたたみ
- 常時最前面のオーバーレイとしてゲーム画面の邪魔にならないサイズ

主要パネル構成：

```
┌──────────────────────────┐
│ タイトルバー（ドラッグ移動）│
├──────────────────────────┤
│ スコアボード（両チーム）    │
├──────────────────────────┤
│ ビルド提案（機能1）        │
├──────────────────────────┤
│ マッチアップTip（機能2）   │
├──────────────────────────┤
│ マクロアドバイス（機能3）   │
├──────────────────────────┤
│ アラート（機能4）          │
└──────────────────────────┘
```

## 5. AI呼び出しの設計方針

### 5.1 いつ呼ぶか（トリガー設計）

AIは遅い（数秒かかる）しコストがかかるので、呼ぶタイミングの制御が重要。

| 機能 | トリガー | 回数 |
|------|---------|------|
| アイテム提案 | アイテム購入検出 / キルデス / 一定時間経過 | 試合中に複数回 |
| マッチアップTip | 試合開始時（ポジション確定後） | 1回 |
| マクロアドバイス | 数十秒間隔 | 試合中に複数回 |
| 試合後コーチング | 試合終了時 | 1回 |

共通して**デバウンス（最低N秒は再呼び出ししない）** をかけ、連続トリガーを抑制する。

### 5.2 何を渡すか（コンテキスト設計）

LLMに渡す情報は「静的」と「動的」に分ける。

- **静的コンテキスト**（試合開始時に1回生成）: チャンピオン構成、チーム特性、統計ベースのコアビルド
- **動的コンテキスト**（毎回更新）: 現在のレベル・アイテム・KDA・ゲーム時間・オブジェクト状態

こうすることで、毎回全データを送り直す無駄を減らせる。

クラウドモード（Bedrock / Claude）では、**Prompt Caching** を活用したコンテキスト設計を行う。静的コンテキスト（システムプロンプト・チャンピオン構成・統計データ等）をキャッシュ対象として先頭に配置し、動的コンテキストを末尾に付加する構成にすることで、キャッシュヒット時のトークン課金を大幅に削減できる。ローカルLLMモードではこの仕組みは不要なため、モードに応じてプロンプト構成を切り替える。

### 5.3 フリーミアムモデルと2つの動作モード

ビジネスモデルはフリーミアム。ローカルLLMモード（無料）とクラウドLLMモード（有料）を提供する。

課金機能は初期リリースでは実装しない。設定画面でFree / Proモードを手動で切り替えられるようにする。

| モード | プロバイダー | 品質 | コスト | 認証 |
|--------|-------------|------|--------|------|
| **ローカル（Free）** | Ollama（ユーザーPC上） | 低〜中。小型モデルの限界あり | 無料 | 不要 |
| **クラウド（Pro）** | AWS Bedrock（開発者AWS） | 高。Claude Haiku | 有料（月額 or 従量） | ビルド時に埋め込み |

#### クラウドモードのアーキテクチャ

```
Electron App  ──AWS SDK──>  Bedrock (Claude)
               (SigV4署名)
```

- ElectronアプリからAWS SDK（`@aws-sdk/client-bedrock-runtime`）でBedrockを直接呼び出す
- サーバーサイドのインフラ（API Gateway / Lambda）は不要
- **Bedrock**: Claude Haiku 4.5 を利用

#### AWS認証情報の管理

- 開発者のAWS Access Key ID / Secret Access Keyをビルド時に `.env` で埋め込む
- 認証情報はElectronのメインプロセス側で保持し、Renderer側には露出させない
- ⚠️ Electronアプリへの認証情報埋め込みはasarの展開等でキーが抽出されるリスクがある。初期段階では許容するが、将来的にはAPI Gateway + Lambda構成への移行を検討する

#### アプリ側のプロバイダー構成

```
AI通信モジュール
├── ローカルLLM接続     ... Ollama経由 (Free)
└── クラウドLLM接続      ... AWS SDK経由でBedrock直接呼び出し (Pro)
```

### 5.4 モデルの使い分け

#### ローカルモード（Free）

全4機能をローカルLLM（Ollama / Qwen3.5 9B相当）で処理する。小型モデルはゲーム固有知識が不正確なため、**前処理でLLMの判断負荷を極力下げる**（候補を絞る、事実情報を事前抽出する等）ことで品質を確保する。

- 2段階パイプライン: Step1自由文分析 → Step2 JSON変換
- LLMにゲーム知識を問わない設計（詳細は9章）

#### クラウドモード（Pro）

全4機能をAWS Bedrock経由のClaudeで処理する。モデルのゲーム知識と推論能力に依存できるため、前処理は最小限でよい。

初期リリースでは全機能をHaikuで統一する。コスト・レイテンシを抑えつつ、前処理層でLLMの負荷を下げる設計と組み合わせることで品質を確保する。

| 機能 | モデル | 理由 |
|------|--------|------|
| アイテム提案 | Haiku | 頻繁に呼ばれる。候補選択タスクなので軽量で十分 |
| マクロアドバイス | Haiku | 同上 |
| マッチアップTip | Haiku | 前処理で統計データ・相性情報を事前抽出するため軽量モデルで対応可能 |
| コーチング | Haiku | 前処理で試合データを構造化し、LLMには文章生成に集中させる |

品質が不十分な機能があれば、必要に応じて上位モデルへの切り替えを検討する。

## 6. データの時系列フロー

```
1. アプリ起動
   → 待機画面。LoLクライアントの接続を監視

2. チャンプセレクト検出
   → チーム構成を表示

3. 試合開始
   → 静的データ取得（チャンプ情報、コアビルド）
   → ポジション確定後、対面チャンプを特定 → マッチアップTip生成（AI、1回）
   → 試合ログ蓄積を開始（コーチング用）
   → ポーリング開始

4. 試合中ループ（数秒間隔）
   → ゲームデータ取得
   → 60秒間隔でスナップショットを試合ログに蓄積（コーチング用）
   → ルールベースアラート評価（即時）
   → 差分検知 → 条件を満たせばAI呼び出し
      → アイテム提案更新
      → マクロアドバイス更新

5. 試合終了
   → 蓄積した試合ログからコーチング用の集計データを生成
   → 試合後コーチング生成（AI、1回）
   → 結果表示 & ローカル保存

6. → 1に戻る
```

## 8. 設計上の重要な判断ポイント

### ゲーム体験を壊さない
- ウィンドウは**常時最前面**だが、LoLの操作を邪魔しないサイズ・位置にする
- CPU/メモリを食いすぎてゲームのFPSを落とさない
- 情報は短く。読ませるのではなく「見せる」

### AI呼び出しの頻度バランス
- 呼びすぎ → コスト爆発、レート制限、応答遅延
- 呼ばなすぎ → アドバイスが古くなって役に立たない
- 「状況が変わったときだけ呼ぶ」が基本戦略

### AIに任せる範囲を絞る
- 統計で答えが出ること（コアビルド等）はAIに聞かない
- タイマー計算（オブジェクトスポーン等）もAIに聞かない
- AIには**「この試合固有の状況判断」** だけを任せる

### オフライン・低コスト対応
- ルールベースアラート（機能4）はAI不要なので、どの環境でも動く
- Ollamaでローカル推論すれば、APIキーなし・課金なしで全機能使える

## 9. AI精度設計

LLMの仕事を **「構造化された事実を、プレイヤーに伝わる自然言語に変換すること」** に限定する。

事実の抽出・候補の生成・数値の比較はすべてコード側で行い、LLMには判断済みの材料だけを渡す。

---

### 9.1 3層パイプライン

```
              生データ (Live Client / OP.GG / Data Dragon)
                │
   ┌────────────▼────────────┐
   │     前処理層              │  事実抽出・タグ化・ルール判定・候補生成
   │                          │  LLM不要。確定的な処理のみ
   └────────────┬────────────┘
                │ 構造化された入力 (JSON)
   ┌────────────▼────────────┐
   │     推論層                │  LLMは候補の優先順位付けと説明文生成だけ担当
   │                          │  入出力ともに固定スキーマ
   └────────────┬────────────┘
                │ 構造化された出力 (JSON)
   ┌────────────▼────────────┐
   │     後処理層              │  禁止表現除去・文字数圧縮・確信度フィルタ
   │                          │  UIに渡す最終形を整形
   └────────────┬────────────┘
                │
            UI表示
```

---

### 9.2 前処理層

**原則: LLMに考えさせず、事実を渡す。**

前処理層はLLMを一切使わない。生データから以下を確定的に計算し、構造化されたJSONとしてまとめる。

#### 9.2.1 統一試合状態オブジェクト

#### データソースの制約と推定ロジック

GameStateの一部のフィールドはAPIから直接取得できないため、推定で算出する。

| 情報 | データソース | 推定方法 |
|------|------------|---------|
| 自分のゴールド | Live Client API | 直接取得可能 |
| 味方/敵チームの推定ゴールド | 推定 | 各プレイヤーの所持アイテム価格を合算。自プレイヤーのみ未使用ゴールドを加算 |
| 育っている/遅れているプレイヤー | 推定 | KDA + 完成アイテム数から判定。ゴールドは不要 |
| レーン状態 | 推定 | 直近のキル/デス/タワー破壊イベントの発生レーンから簡易推定。精度は低いため「不明」をデフォルトとし、判定できた場合のみ値を入れる |
| ドラゴン取得数（味方/敵） | イベントログ | ドラゴンキルイベントのキルしたプレイヤー名からチーム判定。イベントが取得できない場合はカウント不明として扱う |

精度の低い推定値は試合状態オブジェクト上で明示し（「推定値」ラベル、「不明」デフォルト）、LLMへの入力構築時に「確度が低い情報」として扱う。

#### 9.2.2 前処理で確定させること（LLMに任せない）

| 判定項目 | ロジック | 例 |
|---------|---------|-----|
| ゲームフェーズ | 経過時間で判定: 14分未満→序盤、25分未満→中盤、以降→終盤 | 中盤 |
| 戦況 | キル差で判定: ±3以内→互角、+3以上→有利、-3以下→不利 | 互角 |
| 育ち具合判定 | キル5以上かつデス2以下→育っている / デス5以上かつキル1以下→遅れている（フェーズ補正あり） | 育っている |
| 敵ダメージ構成 | アイテムAD/AP + チャンプ基礎ステから比率算出 | AD35%、AP65% |
| 回復阻害の必要性 | 敵の回復持ちが2体以上→必要、3体以上→必須 | 必要 |
| CC量 | 敵チームのCC持ちスキル数をカウント | 多い |
| オブジェクト状態 | イベントログ + ゲーム時間から残り秒を計算 | 次のスポーンまで40秒 |
| チーム構成タイプ | タグ集計: Tank+Engager多→集団戦型、Mage+長射程多→ポーク型 | 集団戦型 |
| 脅威プレイヤー | 育ち具合 + ダメージタイプから特定 | Syndra（育ったAPキャリー） |

#### 9.2.3 機能別の入力構築

味方/敵の構成タイプ      |   △     |     △       |   ○   |    ○     |
候補アイテムリスト       |   ○     |     ×       |   ×   |    ×     |
前回のAI出力             |   ○     |     ×       |   ○   |    ×     |
```

○=必須、△=あれば精度向上、×=渡さない

### 9.3 推論層（LLM）

**原則: LLMの仕事は「候補の優先順位付け」と「自然言語での説明生成」。**

動作モードにより方針が異なる:

- **クラウドモード（Pro）**: Claude HaikuもLoLの詳細なゲーム知識（スキル効果・スケーリング・相性等）を正確に持っている保証はない。そのため、試合に登場するチャンピオンの基礎ステータスやスキル詳細などの前提知識を、Prompt Cachingを活用して初回リクエスト時に静的コンテキストとして渡す。以降のリクエストではキャッシュされた知識を再利用しつつ、動的な試合状況のみを追加する。入力は日本語の構造化JSONをそのまま使える。
- **ローカルモード（Free）**: 小型LLM（9Bクラス）はゲーム知識が不正確なため、LLMにゲーム知識を問わない。前処理が事実情報を完全に抽出し、LLMは文章化のみを担当する。入力は英語ベースの簡潔なテキストで渡す（日本語の長文JSONは小型モデルが正しく解釈できないため）。

#### 9.3.1 入力形式: 構造化JSON

LLMには自由文ではなく、構造化されたJSONで状況を渡す。モデルの読み違いを防ぐため、フィールド名で意味が明確になるようにする。

**アイテム提案の入力例:**

  "enemy_healing": "needed",
  "situation": "even",
  "candidates": [
    { "id": 3156, "name": "マルモティウス", "effect": "MR+50, 魔法シールド", "tag": "counter" },
    { "id": 3072, "name": "ブラッドサースター", "effect": "AD+55, LS+18%, シールド", "tag": "core" },
    { "id": 3033, "name": "モータルリマインダー", "effect": "AD+35, 貫通+25%, 重傷", "tag": "situational" },
    { "id": 3139, "name": "マーキュリーシミター", "effect": "AD+40, MR+30, CC解除", "tag": "counter" }
  ],
  "core_build": ["Infinity Edge", "Phantom Dancer", "Bloodthirster"],
  "previous_advice": { "id": 3156, "reason": "敵AP主体のためMR確保" }

```

チャンプ情報やマクロ知識をLLMに渡す場合、**説明文ではなく構造化された判断材料**として渡す。

```

✅ 構造化された判断材料:
{
  "champion": "Yasuo",
  "danger_skills": ["Q3(ノックアップ)", "W(飛翔体無効化)", "R(ノックアップ連携)"],
  "power_spikes": ["Lv2(EQ)", "Lv6(R)", "2アイテム(IE+PD)"],
  "trade_pattern": "E接近→Q→AA→E離脱",
  "counter_tags": ["CC", "burst", "range_poke"],
  "weakness": "CC、ガンク、序盤レンジ"
}
```

#### 9.3.4 マクロ知識の選択的注入

マクロ戦略の教科書データ（`docs/macro-strategy-prompt.md`）をLLMに渡す際、全文を毎回渡すのではなく、**前処理層がロール・ゲームフェーズに応じて該当セクションだけを抽出**してLLMの入力に含める。

```
前処理層での抽出ロジック:
  1. プレイヤーのロール (TOP/JG/MID/ADC/SUPPORT) → ロール別マクロ責任を抽出
  2. ゲームフェーズ (序盤/中盤/終盤) → 該当フェーズのトレード規則・視界管理を抽出
  3. 戦況 (有利/互角/不利) → 該当する行動指針を抽出

例: role=SUPPORT, gamePhase=mid, situation=behind の場合
  → 「サポートのマクロ責任」+「中盤のトレード規則」+「視界管理の中盤」+「負けている時のマクロ」を抽出
```

抽出されたマクロ知識はロール・チャンピオン情報と同様に**Prompt Cachingの静的コンテキスト**として扱う。ロール別の教科書部分は試合中変わらないため、キャッシュ効率が高い。ゲームフェーズの遷移時（序盤→中盤等）にはキャッシュを更新する。

#### 9.3.5 前回出力のフィードバック

LLMは毎回ゼロから考えるため、前回と矛盾した回答が出やすい。前回の出力を「前回のアドバイス」として毎回渡す。

```json
{
  "previous_advice": {
    "action": "dragon_prep",
    "reason": "ドラゴンソウルリーチ"
  }
}
```

LLMへの指示:
> 前回のアドバイスがある場合、状況が大きく変わっていなければその判断を維持する。変更する場合は変更理由を明示する。

### 9.4 後処理層

**原則: LLMの出力をそのままUIに出さない。**

#### 9.4.1 バリデーション

- JSONスキーマに合致しているか検証。不正なら前回の出力を維持
- 推薦されたアイテムIDが候補リストに含まれているか検証。含まれていなければ除外
- 選択されたアクションがアクション候補リストに含まれているか検証

#### 9.4.2 表現の制御

- 文字数制限: タイトル10文字以内、説明30文字以内、理由1文。超過は切り詰め
- 禁止表現の除去: 「マップを見ろ」「CSを取れ」等の抽象的な一般論を検出した場合は抑制
- 敬語統一: ですます調に統一

#### 9.4.3 確信度フィルタ

**繰り返し実行される機能（アイテム提案・マクロアドバイス）** のLLM出力に確信度を含める。単発実行の機能（マッチアップTip・試合後コーチング）は前回出力が存在しないため、確信度フィルタの対象外とし、代わりにスキーマ検証（9.4.1）とフォールバック（9.5）で品質を担保する。

| 確信度 | 動作 |
|-------|------|
| 高 | そのまま表示 |
| 中 | 表示するが、前回のアドバイスも併記 |
| 低 | 表示しない。前回のアドバイスを維持 |

### 9.5 機能別パイプライン設計

#### アイテム提案

```
前処理:
  ├ 敵ダメージ構成を算出 (AD/AP比率)
  ├ 敵ヒーラー数をカウント → 回復阻害の必要性判定
  ├ 敵CC量を評価 → QSS/マーキュリーの必要性判定
  ├ 自分のロール別ビルド制約を適用 (ADCはコア厳守等)
  └ 候補アイテムリストを生成（最大5個に絞る）:
      1. OP.GG統計のコアビルドから未購入品（最大2個）
      2. カウンターアイテム（敵構成に対する対策品、最大2個）
      3. 状況対応品（回復阻害/QSS等、前処理の判定結果から、最大1個）
      → 各候補に「なぜ候補に入れたか」のタグを付与: core / counter / situational

推論 (LLM):
  └ 候補リスト（最大5個）から最大3つ選択 + 理由 + 次のアクション指示

後処理:
  ├ 選択されたIDが候補リストに存在するか検証
  ├ 前回と同じ推薦の場合はスキップ（UI更新しない）
  └ reasonを1文に圧縮
```

#### マッチアップTip

モデルに依存せず安定した品質を保つため、**前処理でスキル分析を完了**させる。クラウドモード（Haiku）でもローカルモードでも同じ前処理を通すことで、LLMにはゲーム知識を問わない設計を維持する。

```
前処理（コード / LLM不要）:
  ├ Data Dragonのスキル説明テキストからルールベースで特性を抽出:
  │   ├ danger_skills: CC・高バーストを持つスキルを特定（正規表現マッチ）
  │   ├ trade_pattern: 射程・CD・コンボ性からトレードパターンを分類
  │   │   (all_in / short_trade / poke / sustain_lane)
  │   ├ weakness: スキル説明から弱点を推定（CDが長い、マナコストが高い等）
  │   └ counter_tags: CC, burst, sustain, shield 等のタグ付与
  ├ 自チャンプについても同様に特性を抽出
  ├ パワースパイクをチャンプタグ・スキル構成から推定
  ├ マッチアップ難易度をタグの相性から判定
  └ 上記をすべて自然言語の箇条書きに変換（LLMへの入力用）:
      例: "opponent danger: Time Bomb(Q) = AoE + stun when doubled"
          "your strength: Pyromania(passive) = stun every 4 spells"

推論 (LLM):
  └ 前処理が生成した情報を元に、レーニング指針を自然な日本語で文章化
    ※ LLMの役割: 事実の組み合わせと文章化のみ。ゲーム知識の推論は行わない
    ※ クラウドモード: 日本語の構造化JSONをそのまま入力
    ※ ローカルモード: 英語ベースの簡潔なテキストで入力（小型モデルの日本語JSON読解力の限界を回避）。思考機能を無効化して実行

後処理:
  ├ tipsが3つ未満 or JSON不正の場合:
  │   → 前処理の抽出データからテンプレートで簡易Tipを生成（LLM不要のフォールバック）
  │   → 例: danger_skills → 「{skill_name}({key})に注意」
  ├ 出力に含まれるスキル名が入力データに存在するか検証（ハルシネーション検出）
  │   → 入力にないスキル名が含まれていたら該当tipを除去しフォールバックで補完
  └ 各tipを1文に圧縮
```

#### マクロアドバイス

```
前処理:
  ├ オブジェクト状態を算出 (残り秒、取得可能か、ソウルリーチか)
  ├ キル差・ゴールド差から戦況を判定
  ├ レーン状態を判定 (pushing/frozen/enemy_pushing)
  ├ チーム構成タイプから勝利条件を判定 (engage/poke/split/scale)
  ├ 脅威プレイヤーを特定
  └ アクション候補を優先度順に最大3個生成:
      1. オブジェクトが取得可能/準備中 → dragon_secure / dragon_prep / baron_secure / baron_prep
      2. 人数有利（敵デス中） → push_tower / invade
      3. 上記なし → 戦況に応じて farm / recall / ward / split_push から選出
      → 全8種から前処理が3個まで絞り、それぞれに選出理由タグを付与

推論 (LLM):
  └ 候補（最大3個）から1つ選択 + 理由 + 注意点

後処理:
  ├ 選択されたactionが候補リストに存在するか検証
  ├ 前回と同じactionの場合、表示内容は維持し更新時刻のみ更新（見た目を変えない）
  └ title 10文字以内、desc 30文字以内に圧縮
```

#### 試合後コーチング

試合中にポーリングごとのスナップショットをローカルに蓄積し、試合終了時にそれを集計してLLMに渡す。Live Client APIは試合終了後に使えなくなるため、試合中の蓄積が必須。

**試合ログ（試合中に蓄積するデータ）:**

ポーリングごとに以下を配列に追記する（メモリ上に保持、ディスク書き込みは試合終了時のみ）。

```json
{
  "timestamp": 600,
  "kda": [2, 1, 1],
  "cs": 58,
  "items": ["Doran's Blade", "Berserker's Greaves"],
  "level": 8
}
```

全スナップショットを保存する必要はない。**60秒間隔**でサンプリングすれば、30分の試合で30エントリ程度に収まる。

加えて、イベントログ（キル・デス・オブジェクト獲得・タワー破壊）は発生時に記録する。

```
前処理:
  ├ 試合ログの60秒スナップショットからKDA推移・CS推移を集計
  ├ アイテム購入順序（ビルドパス）を復元し、コアビルドとの一致率を算出
  ├ イベントログからオブジェクト獲得時の生死を確認 → オブジェクト参加率を算出
  └ 序盤(0-14分)/中盤(14-25分)/終盤(25分〜)に分割してフェーズ別KDA・CS/分を算出

推論 (LLM):
  └ 集計データを元に、具体的な評価テキストと改善点を生成

後処理:
  ├ score が 1-10 の範囲内か検証
  ├ grade が S/A/B/C/D のいずれかか検証
  └ 改善点が具体的か（抽象論チェック）
```

### 9.6 呼び出し制御

#### トリガー設計

| 機能 | タイプ | トリガー | デバウンス |
|------|-------|---------|-----------|
| マッチアップTip | 単発 | 試合開始時（ポジション確定後） | なし（1回のみ） |
| 試合後コーチング | 単発 | 試合終了時 | なし（1回のみ） |
| アイテム提案 | イベント駆動 | アイテム購入 / キルデス / 一定時間経過 | 10-15秒 |
| マクロアドバイス | 定期+イベント | 一定間隔 + オブジェクトスポーン前 | 30-60秒 |
| ルールベースアラート | 常時 | ポーリングごと | なし（AI不要） |

#### 変化検知

ポーリングごとにGameStateの差分を取り、**意味のある変化**があった場合のみLLMを呼ぶ。

意味のある変化:
- アイテム購入（完成品のみ。素材購入は無視）
- キルまたはデス
- オブジェクト獲得
- ゲームフェーズの遷移
- 一定時間（60秒）経過

#### モデル選択

動作モードによりLLMの能力が異なるため、パイプラインの設計方針をモード別に分ける。

**クラウドモード（Pro）:**

全機能をClaude Haikuで統一する。前処理でゲーム知識（チャンピオン情報・スキル詳細等）をPrompt Cachingで渡し、LLMの知識不足を補う。

| タスクの性質 | モデル | 理由 |
|-------------|--------|------|
| 候補から選択（アイテム・マクロ） | Haiku | 頻繁に呼ばれる。候補選択なので軽量で十分 |
| 自由記述（マッチアップ・コーチング） | Haiku | 前処理で事実を完全に抽出し、Prompt Cachingでゲーム知識を補完 |
| 数値計算・タイマー・タグ付け | コード（LLM不要） | LLMより確実 |

**ローカルモード（Free）:**

小型モデル（9Bクラス）はゲーム知識が不正確なため、LLMの役割を「前処理が用意した候補・事実の選択と文章化」に限定する。

| タスクの性質 | 実行主体 | 理由 |
|-------------|---------|------|
| 候補から選択（アイテム・マクロ） | ローカルLLM | 構造化入力から選ぶだけなので小型モデルで十分 |
| テンプレート穴埋め（マッチアップ・コーチング） | ローカルLLM | 前処理が事実情報を抽出済み。LLMは文章の自然さを担当 |
| 数値計算・タイマー・タグ付け | コード（LLM不要） | LLMより確実 |
| スキル効果の分析・danger_skills抽出 | コード（LLM不要） | 小型LLMはスキル効果を誤認するため、Data Dragonのテキストからルールベースで抽出 |

**小型ローカルLLMの制約と対策:**

| 制約 | 対策 |
|------|------|
| 日本語の長文JSONを正しく読めない | 入力は英語ベース、必要最小限のフィールドに絞る |
| 思考機能でトークンを消費し本文が空になる | 思考機能を無効化する設定を指定 |
| ゲーム固有知識（スキル効果等）が不正確 | LLMにゲーム知識を問わない。事実はすべて前処理で抽出しLLM入力に含める |
| 指示を無視して自前の知識で補完する | 入力データに含まれる情報のみを使うよう明示。出力に入力外のスキル名が含まれていたら後処理で除去 |

## 10. 精度評価

精度改善は、良くなったか測れないと意味がない。

### 10.1 評価基準

機能ごとに以下の4段階で評価する。

| レベル | 定義 | 例 |
|-------|------|-----|
| **誤り** | 事実と矛盾する、または有害なアドバイス | 敵全員ADなのにMRを推奨 |
| **汎用論** | 間違いではないが、この試合の文脈を踏まえていない | 「CSを意識しよう」「マップを見よう」 |
| **文脈的** | この試合の状況を踏まえた具体的なアドバイス | 「敵AP3人なのでMRを優先」 |
| **行動可能** | 今すぐ実行できる具体的な指示 | 「次のリコールでマーキュリーブーツを購入する」 |

目標: **「行動可能」が80%以上、「誤り」が0%。**

### 10.2 評価セット

機能ごとに20-30ケースの評価セットを用意する。各ケースは:

```json
{
  "id": "item_001",
  "scenario": "敵AP3人構成、自分はADC、コア2品完成済み",
  "input": { ... },
  "expected_behavior": "MRアイテム（マルモティウス or ウィッツエンド）を推奨すること",
  "ng_patterns": ["アーマーアイテムを推奨", "攻撃アイテムのみ推奨"]
}
```

### 10.3 回帰テスト

プロンプトやパイプラインを変更するたびに評価セットを実行し、精度が下がっていないことを確認する。確信度「低」の割合が増えた場合も劣化とみなす。
