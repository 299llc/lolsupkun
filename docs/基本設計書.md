# ろるさぽくん — 基本設計書

## 前提条件

- Windows デスクトップアプリ（Electron + React + shadcn/ui + Tailwind CSS）
- LoLの試合中にリアルタイムでAIコーチングを提供する

---

## 1. 解くべき問題と5つの機能

| # | 機能 | 解決する問題 | いつ動く |
|---|------|-------------|---------|
| 1 | アイテム提案 | 敵構成に応じたアイテム選択がわからない | 試合中（状況変化時） |
| 2 | 対面マッチアップTip | 知らない対面で何に気をつけるべきかわからない | チャンプ確定時 |
| 3 | マクロアドバイス | ゲーム中盤以降、どこにいて何をすべきかわからない | 試合中（定期的） |
| 4 | ルールベースアラート | ファーム効率低下やオブジェクトタイミングを見逃す | 試合中（常時） |
| 5 | 試合後コーチング | 何が良くて何が悪かったかわからない | 試合終了後 |

## 2. システム全体像

```
┌───────────────────────────────────────┐
│          ろるさぽくん (Electron)        │
│                                       │
│  ┌───────────┐  IPC  ┌────────────┐  │
│  │ Renderer   │◄─────►│   Main     │  │
│  │ (React UI) │       │ (Backend)  │  │
│  └───────────┘       └─────┬──────┘  │
└────────────────────────────┼─────────┘
                             │
           ┌─────────────────┼─────────────┐
           │                 │             │
  ┌────────▼──────┐  ┌──────▼─────┐ ┌─────▼──────┐
  │ LoLクライアント │  │ AIサービス  │ │ ビルドデータ │
  │ (試合データ)    │  │ (判断生成)  │ │ (統計情報)   │
  └───────────────┘  └────────────┘ └────────────┘
```

アプリは大きく **「データを集める」→「判断を作る」→「画面に出す」** の3段階で動く。

## 3. Main Process の3つの役割

### 3.1 データ収集

LoLクライアントが試合中に公開するローカルAPI（Live Client Data API）から、数秒間隔でゲームデータを取得する。

取得できる情報：
- 全プレイヤーのチャンピオン・レベル・アイテム・KDA・CS
- 自プレイヤーのゴールド（`activePlayer.currentGold`）
- ゲーム内時間
- イベントログ（キル、オブジェクト獲得等）

加えて、LCU（League Client Update）APIからチャンプセレクト中のピック情報も取得する。

#### Live Client Data API で取得できないもの

- **他プレイヤーのゴールド**: 自分以外のゴールドは取得不可。アイテム価格の合算で推定する
- **レーン状態（laneState）**: APIにはウェーブ位置の情報がない。直近のキル/デス/タワー破壊イベントの発生レーンから簡易推定する。精度は低いため、マクロアドバイスでは補助情報として扱う
- **ドラゴン取得チーム**: イベントログにキル情報はあるが、どちらのチームが取ったかは判定が必要。キル直後のマップ位置等から推定する

### 3.2 判断生成

集めたデータを元に、2つの方法でアドバイスを生成する。

**AI分析（機能1, 2, 3, 5）**
- ゲーム状況をテキスト化してLLMに投げ、アドバイスを生成させる
- 毎秒呼ぶわけにはいかないので、「状況が変わったとき」だけ呼ぶ仕組みが必要
- 外部のビルド統計データ（OP.GG等）をベースラインとして渡すことで、AIには「この試合固有の判断」だけに集中させる

**ルールベース判定（機能4）**
- AIを使わない即時判定。閾値ベースの単純なif文
- CS/分が一定以下なら警告、オブジェクトスポーンN秒前に予告、等
- AIの応答を待たずに瞬時に出せるのが強み

### 3.3 アプリ管理

Electron固有の仕事。ウィンドウ管理（フレームレス・常時最前面）、設定の永続化、APIキーの管理など。

## 4. Renderer（React UI）の画面設計

### 4.1 フェーズに応じた画面切替

LoLの試合フローに合わせて画面を切り替える。

```
待機中 ──► チャンプセレクト ──► 試合中 ──► 試合終了
                                          │
                                          └──► 待機中（次の試合へ）
```

| フェーズ | 表示内容 |
|---------|---------|
| 待機中 | 接続状態、設定画面 |
| チャンプセレクト | チーム構成、ルーン・スペル情報 |
| 試合中 | **5機能のうち4つ**（アイテム提案、マッチアップTip、マクロアドバイス、アラート） |
| 試合終了 | 試合後コーチング |

### 4.2 試合中画面のレイアウト方針

「チラ見でわかる」が最重要。ゲーム中に読み込む余裕はない。

- **1つのアドバイスは2-3行以内**に収める
- 優先度の高い情報を上に、補足は折りたたみ
- 常時最前面のオーバーレイとしてゲーム画面の邪魔にならないサイズ

主要パネル構成：

```
┌──────────────────────────┐
│ タイトルバー（ドラッグ移動）│
├──────────────────────────┤
│ スコアボード（両チーム）    │
├──────────────────────────┤
│ ビルド提案（機能1）        │
├──────────────────────────┤
│ マッチアップTip（機能2）   │
├──────────────────────────┤
│ マクロアドバイス（機能3）   │
├──────────────────────────┤
│ アラート（機能4）          │
└──────────────────────────┘
```

## 5. AI呼び出しの設計方針

### 5.1 いつ呼ぶか（トリガー設計）

AIは遅い（数秒かかる）しコストがかかるので、呼ぶタイミングの制御が重要。

| 機能 | トリガー | 回数 |
|------|---------|------|
| アイテム提案 | アイテム購入検出 / キルデス / 一定時間経過 | 試合中に複数回 |
| マッチアップTip | チャンプ確定時 | 1回 |
| マクロアドバイス | 数十秒間隔 | 試合中に複数回 |
| 試合後コーチング | 試合終了時 | 1回 |

共通して**デバウンス（最低N秒は再呼び出ししない）** をかけ、連続トリガーを抑制する。

### 5.2 何を渡すか（コンテキスト設計）

LLMに渡す情報は「静的」と「動的」に分ける。

- **静的コンテキスト**（試合開始時に1回生成）: チャンピオン構成、チーム特性、統計ベースのコアビルド
- **動的コンテキスト**（毎回更新）: 現在のレベル・アイテム・KDA・ゲーム時間・オブジェクト状態

こうすることで、毎回全データを送り直す無駄を減らせる。

### 5.3 プロバイダーの切り替え

AIの呼び先は差し替え可能にする。

| プロバイダー | 特徴 |
|-------------|------|
| Claude API（BYOK） | ユーザーが自分のAPIキーを使う。高品質 |
| Ollama（ローカル） | API キー不要。オフライン動作可能。品質はマシンスペック依存 |

### 5.4 モデルの使い分け

- **速度優先**（アイテム提案・マッチアップ・マクロ）→ 軽量モデル
- **品質優先**（試合後コーチング）→ 高品質モデル

## 6. データの時系列フロー

```
1. アプリ起動
   → 待機画面。LoLクライアントの接続を監視

2. チャンプセレクト検出
   → チーム構成を表示

2.5. チャンプ確定
   → マッチアップTip生成（AI、1回）

3. 試合開始
   → 静的データ取得（チャンプ情報、コアビルド）
   → 試合ログ蓄積を開始（コーチング用）
   → ポーリング開始

4. 試合中ループ（数秒間隔）
   → ゲームデータ取得
   → GameStateスナップショットを試合ログに蓄積
   → ルールベースアラート評価（即時）
   → 差分検知 → 条件を満たせばAI呼び出し
      → アイテム提案更新
      → マクロアドバイス更新

5. 試合終了
   → 蓄積した試合ログからコーチング用の集計データを生成
   → 試合後コーチング生成（AI、1回）
   → 結果表示 & ローカル保存

6. → 1に戻る
```

## 7. 外部依存まとめ

| 依存先 | 何を取るか | いつ使えるか |
|--------|-----------|-------------|
| Live Client Data API (`localhost:2999`) | 試合中のリアルタイムデータ | 試合中のみ |
| LCU API | チャンプセレクト情報、ゲームフロー | LoLクライアント起動中 |
| AI（Claude / Ollama） | アドバイス生成 | 常時（Ollamaはローカル） |
| OP.GG等の統計サイト | コアビルド、カウンターアイテム | 常時 |
| Data Dragon（Riot CDN） | チャンプ名・アイテム名・画像 | 常時 |

## 8. 設計上の重要な判断ポイント

### ゲーム体験を壊さない
- ウィンドウは**常時最前面**だが、LoLの操作を邪魔しないサイズ・位置にする
- CPU/メモリを食いすぎてゲームのFPSを落とさない
- 情報は短く。読ませるのではなく「見せる」

### AI呼び出しの頻度バランス
- 呼びすぎ → コスト爆発、レート制限、応答遅延
- 呼ばなすぎ → アドバイスが古くなって役に立たない
- 「状況が変わったときだけ呼ぶ」が基本戦略

### AIに任せる範囲を絞る
- 統計で答えが出ること（コアビルド等）はAIに聞かない
- タイマー計算（オブジェクトスポーン等）もAIに聞かない
- AIには**「この試合固有の状況判断」** だけを任せる

### オフライン・低コスト対応
- ルールベースアラート（機能4）はAI不要なので、どの環境でも動く
- Ollamaでローカル推論すれば、APIキーなし・課金なしで全機能使える

## 9. AI精度設計

LLMの仕事を **「構造化された事実を、プレイヤーに伝わる自然言語に変換すること」** に限定する。

事実の抽出・候補の生成・数値の比較はすべてコード側で行い、LLMには判断済みの材料だけを渡す。

---

### 9.1 3層パイプライン

```
              生データ (Live Client / OP.GG / Data Dragon)
                │
   ┌────────────▼────────────┐
   │     前処理層              │  事実抽出・タグ化・ルール判定・候補生成
   │     (Preprocessor)       │  LLM不要。確定的な処理のみ
   └────────────┬────────────┘
                │ 構造化された入力 (JSON)
   ┌────────────▼────────────┐
   │     推論層                │  LLMは候補の優先順位付けと説明文生成だけ担当
   │     (LLM)                │  入出力ともに固定スキーマ
   └────────────┬────────────┘
                │ 構造化された出力 (JSON)
   ┌────────────▼────────────┐
   │     後処理層              │  禁止表現除去・文字数圧縮・確信度フィルタ
   │     (Postprocessor)      │  UIに渡す最終形を整形
   └────────────┬────────────┘
                │
            UI表示
```

---

### 9.2 前処理層（Preprocessor）

**原則: LLMに考えさせず、事実を渡す。**

前処理層はLLMを一切使わない。生データから以下を確定的に計算し、構造化されたJSONとしてまとめる。

#### 9.2.1 GameState（統一試合状態オブジェクト）

全データソースを統合した単一のスナップショット。これが全機能の入力の土台になる。

```json
{
  "gameTime": 1120,
  "gamePhase": "mid",
  "situation": "even",

  "me": {
    "champion": "Yasuo",
    "role": "MID",
    "level": 11,
    "kda": [5, 2, 3],
    "cs": 142,
    "items": ["Infinity Edge", "Phantom Dancer"],
    "gold": 800,
    "status": "fed"
  },

  "ally": {
    "kills": 15,
    "estimatedGold": 28000,
    "composition": "engage",
    "hasEngager": true,
    "hasFrontline": true,
    "damageBalance": "mixed",
    "fedPlayers": ["Jinx"]
  },

  "enemy": {
    "kills": 12,
    "estimatedGold": 25000,
    "composition": "poke",
    "damageProfile": { "ad": 35, "ap": 65 },
    "healerCount": 2,
    "ccLevel": "high",
    "fedPlayers": ["Syndra"],
    "behindPlayers": ["Jhin"],
    "threats": [
      { "champion": "Syndra", "reason": "fed_ap_carry", "level": 12, "items": 3 }
    ]
  },

  "objectives": {
    "dragon": { "allyCount": 2, "enemyCount": 1, "nextSpawnSec": 40, "soulPoint": false },
    "baron": { "available": true, "spawnSec": null },
    "herald": { "available": false }
  },

  "laneState": {
    "top": "unknown",
    "mid": "unknown",
    "bot": "unknown"
  }
}
```

#### データソースの制約と推定ロジック

GameStateの一部のフィールドはAPIから直接取得できないため、推定で算出する。

| フィールド | データソース | 推定方法 |
|-----------|------------|---------|
| `me.gold` | Live Client API | `activePlayer.currentGold` で直接取得可能 |
| `ally.estimatedGold` / `enemy.estimatedGold` | 推定 | 各プレイヤーの所持アイテム価格を合算。自プレイヤーのみ `currentGold`（未使用ゴールド）を加算 |
| `fedPlayers` / `behindPlayers` | 推定 | KDA + 完成アイテム数から判定。ゴールドは不要 |
| `laneState` | 推定 | 直近のキル/デス/タワー破壊イベントの発生レーンから簡易推定。精度は低いため `"unknown"` をデフォルトとし、判定できた場合のみ値を入れる |
| `objectives.allyCount` / `enemyCount` | イベントログ | DragonKillイベントのKillerNameからチーム判定。イベントが取得できない場合はカウント不明として扱う |

精度の低い推定値はGameState上で明示し（`estimated` prefix、`unknown` デフォルト）、LLMへの入力構築時に「確度が低い情報」として扱う。

#### 9.2.2 前処理で確定させること（LLMに任せない）

| 判定項目 | ロジック | 例 |
|---------|---------|-----|
| ゲームフェーズ | `time < 840 → early, < 1500 → mid, else → late` | `"mid"` |
| 戦況 | キル差で判定: `±3 → even, +3~ → ahead, -3~ → behind` | `"even"` |
| fed/behind判定 | `kills ≥ 5 && deaths ≤ 2 → fed` / `deaths ≥ 5 && kills ≤ 1 → behind`（フェーズ補正あり） | `"fed"` |
| 敵ダメージ構成 | アイテムAD/AP + チャンプ基礎ステから比率算出 | `{ ad: 35, ap: 65 }` |
| 回復阻害の必要性 | 敵healerCount ≥ 2 → `needed`, ≥ 3 → `required` | `"needed"` |
| CC量 | 敵チームのCC持ちスキル数をカウント | `"high"` |
| オブジェクト状態 | イベントログ + ゲーム時間から残り秒を計算 | `{ nextSpawnSec: 40 }` |
| チーム構成タイプ | タグ集計: Tank+Engager多 → `engage`, Mage+長射程多 → `poke` | `"engage"` |
| 脅威プレイヤー | fed判定 + ダメージタイプから特定 | `[{ champion: "Syndra", reason: "fed_ap_carry" }]` |

#### 9.2.3 機能別の入力構築

前処理層はGameStateから **機能ごとに必要な情報だけを抽出** してLLMへの入力JSONを構築する。全データを毎回渡さない。

```
                      | アイテム | マッチアップ | マクロ | コーチング |
─────────────────────┼─────────┼─────────────┼───────┼──────────┤
me（自分の状態）       |   ○     |     ○       |   ○   |    ○     |
enemy.damageProfile   |   ○     |     ×       |   ×   |    ○     |
enemy.threats         |   ○     |     ×       |   ○   |    ○     |
enemy.healerCount     |   ○     |     ×       |   ×   |    ○     |
対面チャンプ詳細       |   △     |     ○       |   ×   |    ○     |
objectives            |   ×     |     ×       |   ○   |    ○     |
laneState             |   ×     |     ×       |   ○   |    △     |
ally/enemy構成タイプ   |   △     |     △       |   ○   |    ○     |
候補アイテムリスト      |   ○     |     ×       |   ×   |    ×     |
前回のAI出力          |   ○     |     ×       |   ○   |    ×     |
```

○=必須、△=あれば精度向上、×=渡さない

### 9.3 推論層（LLM）

**原則: LLMの仕事は「候補の優先順位付け」と「自然言語での説明生成」だけ。**

```json
  "good_points": ["CS安定", "集団戦での位置取り"],
  "improve_points": ["ワード設置頻度", "オブジェクト前のリコールタイミング"],
  "next_game_advice": "オブジェクトスポーン60秒前にはリコールを完了させる意識を持つ"
}
```

#### 9.3.3 LLMに渡す知識（RAG）

チャンプ情報やマクロ知識をLLMに渡す場合、**説明文ではなく構造化された判断材料**として渡す。

```
❌ 長い説明文:
「ヤスオはクリティカル率が2倍になるパッシブを持ち、風の壁で飛翔体を防げます。
 コアビルドはIE→PD→BTで、Lv6以降のオールインが強力です...（200文字）」

✅ 構造化された判断材料:
{
  "champion": "Yasuo",
  "danger_skills": ["Q3(ノックアップ)", "W(飛翔体無効化)", "R(ノックアップ連携)"],
  "power_spikes": ["Lv2(EQ)", "Lv6(R)", "2アイテム(IE+PD)"],
  "trade_pattern": "E接近→Q→AA→E離脱",
  "counter_tags": ["CC", "burst", "range_poke"],
  "weakness": "CC、ガンク、序盤レンジ"
}
```

#### 9.3.4 前回出力のフィードバック

LLMは毎回ゼロから考えるため、前回と矛盾した回答が出やすい。前回の出力を `previous_advice` フィールドとして毎回渡す。

```json
{
  "previous_advice": {
    "action": "dragon_prep",
    "reason": "ドラゴンソウルリーチ"
  }
}
```

LLMへの指示:
> `previous_advice` がある場合、状況が大きく変わっていなければその判断を維持する。変更する場合は変更理由を明示する。

### 9.4 後処理層（Postprocessor）

**原則: LLMの出力をそのままUIに出さない。**

#### 9.4.1 バリデーション

- JSONスキーマに合致しているか検証。不正なら前回の出力を維持
- `recommended` のアイテムIDが候補リストに含まれているか検証。含まれていなければ除外
- `action` がアクション候補リストに含まれているか検証

#### 9.4.2 表現の制御

- 文字数制限: `title` 10文字以内、`desc` 30文字以内、`reason` 1文。超過は切り詰め
- 禁止表現の除去: 「マップを見ろ」「CSを取れ」等の抽象的な一般論を検出した場合は抑制
- 敬語統一: ですます調に統一

#### 9.4.3 確信度フィルタ

LLMの出力に `confidence` フィールドを含める。

| confidence | 動作 |
|-----------|------|
| `high` | そのまま表示 |
| `medium` | 表示するが、前回のアドバイスも併記 |
| `low` | 表示しない。前回のアドバイスを維持 |

### 9.5 機能別パイプライン設計

#### アイテム提案

```
前処理:
  ├ 敵ダメージ構成を算出 (AD/AP比率)
  ├ 敵ヒーラー数をカウント → 回復阻害の必要性判定
  ├ 敵CC量を評価 → QSS/マーキュリーの必要性判定
  ├ 自分のロール別ビルド制約を適用 (ADCはコア厳守等)
  └ 候補アイテムリストを生成（最大5個に絞る）:
      1. OP.GG統計のコアビルドから未購入品（最大2個）
      2. カウンターアイテム（敵構成に対する対策品、最大2個）
      3. 状況対応品（回復阻害/QSS等、前処理の判定結果から、最大1個）
      → 各候補に「なぜ候補に入れたか」のタグを付与: core / counter / situational

推論 (LLM):
  └ 候補リスト（最大5個）から最大3つ選択 + 理由 + 次のアクション指示

後処理:
  ├ 選択されたIDが候補リストに存在するか検証
  ├ 前回と同じ推薦の場合はスキップ（UI更新しない）
  └ reasonを1文に圧縮
```

#### マッチアップTip

```
前処理:
  ├ 自チャンプ・対面チャンプの構造化データを生成
  │   (danger_skills, power_spikes, trade_pattern, counter_tags, weakness)
  └ マッチアップ難易度を統計データから付与

推論 (LLM):
  └ 構造化データを元に、このマッチアップ固有のレーニング指針を生成

後処理:
  ├ tipsが3つ未満 or JSON不正の場合:
  │   → 構造化データからテンプレートで簡易Tipを生成（LLM不要のフォールバック）
  │   → 例: danger_skills → 「{skill}に注意」、power_spikes → 「{spike}が敵のスパイク」
  └ 各tipを1文に圧縮
```

#### マクロアドバイス

```
前処理:
  ├ オブジェクト状態を算出 (残り秒、取得可能か、ソウルリーチか)
  ├ キル差・ゴールド差から戦況を判定
  ├ レーン状態を判定 (pushing/frozen/enemy_pushing)
  ├ チーム構成タイプから勝利条件を判定 (engage/poke/split/scale)
  ├ 脅威プレイヤーを特定
  └ アクション候補を優先度順に最大3個生成:
      1. オブジェクトが取得可能/準備中 → dragon_secure / dragon_prep / baron_secure / baron_prep
      2. 人数有利（敵デス中） → push_tower / invade
      3. 上記なし → 戦況に応じて farm / recall / ward / split_push から選出
      → 全8種から前処理が3個まで絞り、それぞれに選出理由タグを付与

推論 (LLM):
  └ 候補（最大3個）から1つ選択 + 理由 + 注意点

後処理:
  ├ 選択されたactionが候補リストに存在するか検証
  ├ 前回と同じactionの場合、表示内容は維持し更新時刻のみ更新（見た目を変えない）
  └ title 10文字以内、desc 30文字以内に圧縮
```

#### 試合後コーチング

試合中にポーリングごとのスナップショットをローカルに蓄積し、試合終了時にそれを集計してLLMに渡す。Live Client APIは試合終了後に使えなくなるため、試合中の蓄積が必須。

**試合ログ（試合中に蓄積するデータ）:**

ポーリングごとに以下を配列に追記する（メモリ上に保持、ディスク書き込みは試合終了時のみ）。

```json
{
  "timestamp": 600,
  "kda": [2, 1, 1],
  "cs": 58,
  "items": ["Doran's Blade", "Berserker's Greaves"],
  "level": 8
}
```

全スナップショットを保存する必要はない。**60秒間隔**でサンプリングすれば、30分の試合で30エントリ程度に収まる。

加えて、イベントログ（キル・デス・オブジェクト獲得・タワー破壊）は発生時に記録する。

```
前処理:
  ├ 試合ログの60秒スナップショットからKDA推移・CS推移を集計
  ├ アイテム購入順序（ビルドパス）を復元し、コアビルドとの一致率を算出
  ├ イベントログからオブジェクト獲得時の生死を確認 → オブジェクト参加率を算出
  └ 序盤(0-14分)/中盤(14-25分)/終盤(25分〜)に分割してフェーズ別KDA・CS/分を算出

推論 (LLM):
  └ 集計データを元に、具体的な評価テキストと改善点を生成

後処理:
  ├ score が 1-10 の範囲内か検証
  ├ grade が S/A/B/C/D のいずれかか検証
  └ 改善点が具体的か（抽象論チェック）
```

### 9.6 呼び出し制御

#### トリガー設計

| 機能 | タイプ | トリガー | デバウンス |
|------|-------|---------|-----------|
| マッチアップTip | 単発 | チャンプ確定時 | なし（1回のみ） |
| 試合後コーチング | 単発 | 試合終了時 | なし（1回のみ） |
| アイテム提案 | イベント駆動 | アイテム購入 / キルデス / 一定時間経過 | 10-15秒 |
| マクロアドバイス | 定期+イベント | 一定間隔 + オブジェクトスポーン前 | 30-60秒 |
| ルールベースアラート | 常時 | ポーリングごと | なし（AI不要） |

#### 変化検知

ポーリングごとにGameStateの差分を取り、**意味のある変化**があった場合のみLLMを呼ぶ。

意味のある変化:
- アイテム購入（完成品のみ。素材購入は無視）
- キルまたはデス
- オブジェクト獲得
- ゲームフェーズの遷移
- 一定時間（60秒）経過

#### モデル選択

| タスクの性質 | 適切なモデル | 理由 |
|-------------|------------|------|
| 候補から選択（アイテム・マクロ） | 軽量・高速モデル | 構造化入力から選ぶだけなので軽量で十分 |
| 自由記述（マッチアップ・コーチング） | 高品質モデル | 文脈を踏まえた文章生成が必要 |
| 数値計算・タイマー・タグ付け | コード（LLM不要） | LLMより確実 |

## 10. 精度評価

精度改善は、良くなったか測れないと意味がない。

### 10.1 評価基準

機能ごとに以下の4段階で評価する。

| レベル | 定義 | 例 |
|-------|------|-----|
| **誤り** | 事実と矛盾する、または有害なアドバイス | 敵全員ADなのにMRを推奨 |
| **汎用論** | 間違いではないが、この試合の文脈を踏まえていない | 「CSを意識しよう」「マップを見よう」 |
| **文脈的** | この試合の状況を踏まえた具体的なアドバイス | 「敵AP3人なのでMRを優先」 |
| **行動可能** | 今すぐ実行できる具体的な指示 | 「次のリコールでマーキュリーブーツを購入する」 |

目標: **「行動可能」が80%以上、「誤り」が0%。**

### 10.2 評価セット

機能ごとに20-30ケースの評価セットを用意する。各ケースは:

```json
{
  "id": "item_001",
  "scenario": "敵AP3人構成、自分はADC、コア2品完成済み",
  "input": { ... },
  "expected_behavior": "MRアイテム（マルモティウス or ウィッツエンド）を推奨すること",
  "ng_patterns": ["アーマーアイテムを推奨", "攻撃アイテムのみ推奨"]
}
```

### 10.3 回帰テスト

プロンプトやパイプラインを変更するたびに評価セットを実行し、精度が下がっていないことを確認する。`confidence: low` の割合が増えた場合も劣化とみなす。
